Introduction & Concept

In modern web development, monolithic models are increasingly replaced by decoupled headless architectures. In this design paradigm, the data storage logic (Backend) is separated from the UI presentation layer (Frontend). The E-Sphere project implements a robust template of this modern stack.

By routing the backend on Laravel 11 and the frontend as a single-page application (SPA) on React (Vite), E-Sphere balances fast speeds with high-grade security.

This book serves as a developer blueprint. It outlines the architecture, explains database mappings, details Redux state and token interceptor patterns, and breaks down the core AI chatbot architecture powered by Google Gemini 2.5 Flash.

1. The Core System Architecture

The interaction between the React user interface and the Laravel server database works on a strictly stateless HTTP communication cycle. When a user requests data, the React client initiates an Axios API call to the Laravel server. Laravel queries the MySQL database and feeds the response back as a JSON payload.

B. Security and CORS (Cross-Origin Resource Sharing)

Because the frontend (React SPA) and backend (Laravel REST API) run on completely separate servers (usually on separate ports), CORS rules must be configured in Laravel to allow Vite origins. Laravel 11 handles this internally inside the config/cors.php configuration to allow cross-origin requests:

  • paths set to target ['api/*', 'sanctum/csrf-cookie'].
  • allowed_methods set to allow ['*'].
  • allowed_origins matches the local Vite host ['http://localhost:5173'].
  • supports_credentials is set to true.

2. Backend Database & REST API

The backend serves as a secure data source. Built on Laravel 11, it houses custom models, Eloquent relations, and controllers that perform CRUD processes and route requests.

A. Eloquent Database Models

Here is how the core Product.php model is implemented. Note the use of casts to convert SQL fields to structured PHP types automatically: 

<?php

namespace App\Models;

use Illuminate\Database\Eloquent\Factories\HasFactory;
use Illuminate\Database\Eloquent\Model;

class Product extends Model
{
    use HasFactory;

    protected $fillable = [
        'category_id', 'brand_id', 'name', 'slug', 'description', 
        'specifications', 'price', 'discount_price', 'stock', 
        'sku', 'is_featured', 'is_active', 'rating'
    ];

    protected $casts = [
        'specifications' => 'array',
        'is_featured' => 'boolean',
        'is_active' => 'boolean',
        'price' => 'decimal:2',
        'discount_price' => 'decimal:2',
        'rating' => 'decimal:2',
    ];

    public function category()
    {
        return $this->belongsTo(Category::class);
    }

    public function brand()
    {
        return $this->belongsTo(Brand::class);
    }

    public function reviews()
    {
        return $this->hasMany(Review::class)->where('is_approved', true);
    }
}

B. Cart Merging Database Logic

In many legacy platforms, if a guest user logs in, their previous shopping selections are wiped. In E-Sphere, a custom backend routine in CartController.php intercepts logins. It loops through the guest's browser cart token contents, checks if the items already exist in the logged-in customer's database row, combines the quantities, and updates the SQL database.

3. Frontend State & Interceptors

React handles user interactions. All pages dynamically load inside the React Router DOM single-page lifecycle, calling APIs behind the scenes without ever triggering browser page refreshes.

A. Global Redux State (Cart Slice)

Redux manages cart item states in real-time, refreshing badge counts in the header on every mutation: 

// frontend/src/store/cartSlice.js
import { createSlice, createAsyncThunk } from '@reduxjs/toolkit';
import api from '../utils/api';

export const fetchCart = createAsyncThunk('cart/fetchCart', async () => {
  const response = await api.get('/cart');
  return response.data;
});

const cartSlice = createSlice({
  name: 'cart',
  initialState: { items: [], loading: false, totalAmount: 0 },
  reducers: {},
  extraReducers: (builder) => {
    builder
      .addCase(fetchCart.fulfilled, (state, action) => {
        state.items = action.payload.items;
        state.totalAmount = action.payload.total;
        state.loading = false;
      });
  }
});

export default cartSlice.reducer;

B. Token-based Session Interceptor

The frontend requests database assets through api.js. If the user logs in, the authentication token generated by Laravel Sanctum is stored in the browser's localStorage and automatically appended to the Authorization: Bearer <Token> headers on all subsequent REST calls.

B. Secure Database Grounding (Avoiding Fake Recommendations)

In ChatbotController.php, we query active database catalog products and the user's order history. By structuring the prompt with this data, we anchor the LLM, forcing it to recommend actual inventory and refer to actual orders: 

// Prompt Context Grounding logic snippet
$products = Product::where('is_active', true)
    ->select('id', 'name', 'slug', 'price', 'discount_price', 'rating', 'stock')
    ->limit(80)->get();

$catalogContext = "";
foreach ($products as $p) {
    $catalogContext .= "- **{$p->name}** | Slug: {$p->slug} | Price: ₹{$p->price}\n";
}

// User context order injection (Only if user has active Sanctum Session)
$user = $request->user('sanctum');
$userContext = $user ? "Orders for {$user->name}: " . $user->orders()->latest()->limit(5)->get()->toJson() : "Guest User";

C. Routing Link Parser (SPA Navigation Helper)

In React, if the AI outputs a standard markdown link [My Product](/product/slug), normal rendering would use an anchor tag <a href="...">. This would trigger a full page reload, breaking the SPA flow.

E-Sphere fixes this by including a custom rendering function in Chatbot.jsx. This function matches markdown links, checks if they start with a relative path (e.g. /product), and swaps them for React Router's <Link> tags.

5. Database Schema & Relations

E-Sphere runs on MySQL on port 3307. The database tables are defined using Laravel database migrations.

A. Table Mappings & Key Relationships

  • users: Holds user details, passwords (hashed), and profile icons.
  • categories & brands: Organized hierarchically. Categories support parent-child relationships (e.g., Electronics ➔ Mobiles).
  • products: Holds items. Linked to category_id and brand_id. Holds stock counts.
  • orders: Linked to user_id and address_id. Contains totals, shipping charges, and tracking states (e.g., pending, processing, shipped, delivered).
  • order_items: Relational pivot table linking order_id and product_id. Records historical product pricing at checkout to protect sales ledger data.

6. Comprehensive Setup Guide

To clone, build, configure environment values, and run the E-Sphere platform, run the following processes sequentially.

A. Clone & Backend Environment Setup

# Clone repository
git clone https://github.com/sonuukush/laravelreactecom.git
cd laravelreactecom/backend

# Install package dependencies
composer install

# Duplicate environmental placeholder
copy .env.example .env

Open the newly created .env file. Configure the database connector variables (note that the local MySQL instance runs on port 3307 by default). Additionally, add your Google Gemini API Key: 

DB_CONNECTION=mysql
DB_HOST=127.0.0.1
DB_PORT=3307
DB_DATABASE=ecom_db
DB_USERNAME=root
DB_PASSWORD=

GEMINI_API_KEY=your_google_gemini_api_key_here

B. Database Migration and Seeding

With the configuration complete, generate the security key and seed mock database records: 

# Generate security App Key
php artisan key:generate

# Build database schema tables and seed catalogs
php artisan migrate --seed

# Create storage link for dynamic images
php artisan storage:link

# Spin up backend server
php artisan serve

C. Frontend Installation and Run

Open a new command line prompt, navigate to the frontend directory, install JavaScript dependencies, and run the Vite server: 

cd ../frontend

# Install node dependencies
npm install

# Run the dev server
npm run dev

Vite will build local modules and boot the storefront at http://localhost:5173. The chatbot will be visible as a floating circle in the bottom right corner.
